# 交通应急指挥大模型 · Skill-Based Agent 架构技术方案

> 版本：v1.0 | 日期：2026年4月 | 定位：详细设计文档 · 面向开发团队

---

## 1. 概述与目标

### 1.1 项目背景

当前系统已具备应急指挥智能体的核心雏形，包括 Agent 调度、知识检索、地图感知、多模态理解、风险评估等能力。但当前存在以下核心问题：

- 所有工具定义全部塞入 system prompt，随着工具增多，token 消耗和模型注意力稀释问题严重
- 缺少任务编排能力，模型是被动响应而不是主动推进
- 无标准化的应急指挥流程框架，方案输出质量不稳定
- 人机协作机制缺失，模型不知道什么时候该停下来问人

### 1.2 设计目标

将系统从"被动问答"升级为"任务驱动的自主 Agent"——接收灾情后能够自主推进完整的应急指挥流程。核心设计目标如下：

1. 引入 Skill 机制，按任务阶段动态加载工具，替代全量工具塞入 prompt 的方式
2. 建立标准化应急指挥流程（SOP），由主流程 Skill 编排各阶段任务
3. 实现主动式人机协作：缺信息时主动询问，不确定时主动提供选项
4. 保持架构的可扩展性，新增 Skill 不需要修改核心调度代码

### 1.3 类比说明

可以把这个系统类比为"应急指挥场景的 Claude Code"——用户给出一个灾情描述，Agent 就像一个经验丰富的指挥员，主动收集信息、调配资源、制定方案、征求确认，最终输出一份完整的指挥方案。

## 2. 整体架构

### 2.1 架构分层

系统分为四层，从上到下依次为：

| 层次 | 组件 | 职责 |
|------|------|------|
| 交互层 | Web / CLI | 用户输入灾情、接收方案、响应人机交互 |
| 编排层 | Agent Core + Skill Router | 状态机驱动，管理任务阶段过渡，动态加载 Skill |
| 能力层 | 6 个 Skills | 各自封装工具集、prompt 片段、输出规范 |
| 基础层 | LLM / RAG / 外部API / 数据 | 模型推理、向量检索、高德接口、知识库 |

### 2.2 核心设计原则

- **Skill 为一等公民：** 每个 Skill 是独立的能力单元，包含工具定义、prompt 片段、输出规范，新增/移除 Skill 不需修改 Agent Core
- **状态机驱动：** Agent 根据当前任务阶段决定激活哪些 Skill，而不是每次都加载全部工具
- **主动推进：** Agent 接收灾情后自主执行完整流程，不依赖用户逐步提问
- **人在回路：** 关键决策点必须人工确认，紧急情况可先给默认方案再征求确认

### 2.3 Skill 与任务阶段的映射关系

下表展示了 6 个 Skill 在各任务阶段的参与情况：

| 任务阶段 | 主流程 | 态势感知 | 知识检索 | 资源调度 | 风险评估 | 人机协作 |
|----------|--------|----------|----------|----------|----------|----------|
| 灾情接收 | ✓ 主控 | | | | | ✓ 补问 |
| 信息补全 | ✓ 编排 | ✓ 主力 | | | | ✓ 补问 |
| 方案生成 | ✓ 主控 | ✓ 辅助 | ✓ 主力 | ✓ 主力 | | ✓ 选项 |
| 方案评估 | ✓ 编排 | | | | ✓ 主力 | ✓ 确认 |
| 方案输出 | ✓ 主控 | | | | | |

## 3. 任务状态机设计

### 3.1 状态定义

Agent 的整个执行过程由一个状态机驱动。每个状态对应一个明确的任务阶段：

| 状态名 | 英文标识 | 说明 | 激活的 Skill |
|--------|----------|------|-------------|
| 灾情接收 | INTAKE | 解析用户输入，提取结构化灾情信息 | 主流程、人机协作 |
| 信息补全 | SITUATIONAL_AWARENESS | 补全地理、天气、路况、现场媒体等信息 | 主流程、态势感知、人机协作 |
| 方案生成 | PLAN_GENERATION | 检索法规案例、盘点资源、生成处置方案 | 主流程、知识检索、资源调度、态势感知、人机协作 |
| 方案评估 | PLAN_EVALUATION | 对生成方案进行多维度评估 | 主流程、风险评估、人机协作 |
| 方案输出 | OUTPUT | 整合输出最终指挥方案 | 主流程 |
| 等待用户 | WAITING_USER | 挂起等待用户回复（补信息/选方案/确认） | 人机协作 |

### 3.2 状态过渡规则

状态过渡由主流程 Skill 的 prompt 引导模型决策，通过结构化输出实现：

| 当前状态 | 过渡条件 | 下一状态 |
|----------|----------|----------|
| INTAKE | 灾情信息完整（类型、位置、严重程度均已知） | SITUATIONAL_AWARENESS |
| INTAKE | 缺少关键信息 | WAITING_USER |
| SITUATIONAL_AWARENESS | 环境信息已补全 | PLAN_GENERATION |
| SITUATIONAL_AWARENESS | 需要用户提供额外信息（如图片、视频） | WAITING_USER |
| PLAN_GENERATION | 方案已生成 | PLAN_EVALUATION |
| PLAN_GENERATION | 存在多个可行方案，需用户选择 | WAITING_USER |
| PLAN_EVALUATION | 评估通过 | OUTPUT |
| PLAN_EVALUATION | 评估发现重大风险，需调整 | PLAN_GENERATION |
| PLAN_EVALUATION | 需用户确认风险接受度 | WAITING_USER |
| WAITING_USER | 用户回复后，根据上下文回到对应阶段 | 返回之前的阶段 |

### 3.3 状态数据结构

Agent 在运行期间维护一个 TaskState 对象，记录当前任务的全部状态信息：

```python
class TaskState:
    current_phase: str                    # 当前阶段标识
    incident_info: IncidentInfo           # 结构化灾情信息
    environment_info: EnvironmentInfo     # 环境、天气、路况等
    available_resources: list             # 已查询到的可用资源
    knowledge_refs: list                  # 检索到的法规/案例引用
    candidate_plans: list                 # 候选方案列表
    evaluation_results: list              # 评估结果
    pending_question: PendingQuestion     # 当前等待用户回答的问题
    conversation_history: list            # 会话历史
    tool_call_log: list                   # 工具调用记录
```

## 4. Skill 系统设计

### 4.1 Skill 抽象定义

每个 Skill 是一个独立的能力单元，包含以下四个组成部分：

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| 元信息 (metadata) | Skill 名称、描述、版本、适用阶段 | name: situational_awareness, phases: [INTAKE, SITUATIONAL_AWARENESS] |
| 工具集 (tools) | 该 Skill 包含的工具列表及其 Function Calling Schema | geocode_address, check_traffic_status, get_weather_by_location |
| Prompt 片段 (prompt_fragment) | 激活时注入 system prompt 的指导文本 | "你现在需要补全环境信息，优先查询天气和路况…" |
| 输出规范 (output_schema) | 该 Skill 工具返回结果的结构化格式要求 | JSON Schema 或文本模板 |

### 4.2 Skill 文件结构

每个 Skill 对应一个目录，包含以下文件：

```
skills/
  situational_awareness/
    SKILL.yaml          # 元信息 + 配置
    prompt.md           # 注入 system prompt 的指导文本
    tools.py            # 工具实现代码
    schema.json         # 输出规范
```

SKILL.yaml 示例结构：

```yaml
name: situational_awareness
description: 态势感知能力，负责补全环境信息
version: 1.0
active_phases:
  - INTAKE
  - SITUATIONAL_AWARENESS
  - PLAN_GENERATION
tools:
  - geocode_address
  - reverse_geocode
  - check_traffic_status
  - get_weather_by_location
  - media_caption
  - search_nearby_pois
priority: high
dependencies: []
```

### 4.3 Skill 加载与卸载机制

Skill Router 在每次状态过渡时执行以下操作：

1. 读取当前状态的 phase 标识
2. 扫描所有 Skill 的 active_phases 字段，筛选出当前阶段应激活的 Skill 列表
3. 合并被激活 Skill 的工具定义，生成当前轮次的 tools 参数
4. 合并被激活 Skill 的 prompt 片段，追加到 system prompt 末尾
5. 调用模型，仅传入当前激活的工具

这样做的核心价值：模型在任何一个时刻只看到当前阶段需要的工具，而不是全部工具。这显著降低了 token 消耗和模型注意力稀释。

## 5. 六大 Skill 详细设计

### 5.1 主流程 Skill（Master Flow）

| 属性 | 内容 |
|------|------|
| 名称 | master_flow |
| 定位 | 整个应急指挥任务的"大脑"，负责流程编排与阶段推进 |
| 激活阶段 | 全部阶段（常驻） |
| 包含工具 | 无直接工具，通过 prompt 指导模型调用其他 Skill 的工具 |

#### 5.1.1 核心职责

- 定义应急指挥 SOP（标准作业流程）
- 在每个阶段开始时，通过 prompt 告诉模型当前处于哪个阶段、目标是什么、可用的工具有哪些
- 维护结构化灾情信息（IncidentInfo），跟踪信息完备程度
- 判断状态过渡时机
- 定义最终方案的输出格式

#### 5.1.2 灾情信息结构化模型（IncidentInfo）

主流程 Skill 的第一任务是将用户自然语言描述转化为结构化灾情信息：

| 字段 | 类型 | 是否必填 | 说明 |
|------|------|----------|------|
| incident_type | string | ✓ | 事故类型（车祸、危化品泄漏、地质灾害、火灾、洪涝等） |
| severity | enum | ✓ | 严重程度：low / medium / high / critical |
| location_text | string | ✓ | 用户描述的位置文本 |
| location_coords | object | 可补全 | 经纬度，通过 geocode 工具补全 |
| time | string | 可推断 | 事发时间，默认当前时间 |
| casualties | object | | 伤亡情况：死亡/受伤/被困人数 |
| hazmat_involved | boolean | | 是否涉及危化品 |
| hazmat_type | string | | 危化品类型（若涉及） |
| road_info | string | | 道路信息：高速/国道/市区道路、车道数等 |
| vehicles_involved | string | | 涉事车辆信息 |
| additional_context | string | | 其他补充信息 |

#### 5.1.3 完备性检查规则

主流程 Skill 在 INTAKE 阶段结束时检查以下必要信息是否已知：

- incident_type 必须明确
- severity 必须明确或可推断
- location_text 或 location_coords 至少有一个

如果任何必要信息缺失，进入 WAITING_USER 状态，通过人机协作 Skill 向用户询问。

#### 5.1.4 最终方案输出格式

在 OUTPUT 阶段，主流程 Skill 要求模型按以下结构输出最终方案：

```json
{
  "事件概述": { "..." },
  "指挥架构": { "总指挥", "副指挥", "各组职责" },
  "处置步骤": [
    { "阶段", "行动", "责任人", "时间要求" }
  ],
  "资源调度": [
    { "资源类型", "名称", "距离", "预计到达", "联系人" }
  ],
  "风险提示": [ "..." ],
  "法规依据": [
    { "来源", "条款", "摘要" }
  ],
  "评估结果": { "综合得分", "风险等级", "改进建议" }
}
```

### 5.2 态势感知 Skill（Situational Awareness）

| 属性 | 内容 |
|------|------|
| 名称 | situational_awareness |
| 定位 | 搜集和补全现场环境信息，为方案生成提供事实基础 |
| 激活阶段 | INTAKE, SITUATIONAL_AWARENESS, PLAN_GENERATION |
| 包含工具 | geocode_address, reverse_geocode, check_traffic_status, get_weather_by_location, media_caption, search_nearby_pois |

#### 5.2.1 工具详细说明

| 工具名 | 输入 | 输出 | 业务场景 |
|--------|------|------|----------|
| geocode_address | 地址文本 | 经纬度 | 将用户描述的位置转为坐标，为后续工具提供坐标输入 |
| reverse_geocode | 经纬度 | 可读地址 | 将设备上报坐标转为可读地址 |
| check_traffic_status | 经纬度 + 半径 | 拥堵指数 + 路段详情 | 评估事故对周边交通的影响，辅助救援路线规划 |
| get_weather_by_location | 经纬度 | 天气状况 + 预警 | 雨雪雾等恶劣天气直接影响处置策略 |
| media_caption | 图片/视频路径 | 场景描述 + 风险点 + 关键信息 | 将现场图片/视频转化为模型可理解的文本 |
| search_nearby_pois | 经纬度 + 类型 | POI 列表 | 搜索公开兴趣点，作为内部资源不足时的补充 |

#### 5.2.2 Prompt 片段要点

- 优先用 geocode_address 把位置文本转为坐标，这是后续多个工具的前置依赖
- 天气和路况查询应并行发起（Web 端已支持同轮多工具调用）
- 如果用户上传了图片/视频，必须先调用 media_caption
- search_nearby_pois 仅在内部资源不足时使用，不应作为首选

---

### 5.3 知识检索 Skill（Knowledge Retrieval）

| 属性 | 内容 |
|------|------|
| 名称 | knowledge_retrieval |
| 定位 | 从法规、预案、技术指南、历史案例中检索相关依据 |
| 激活阶段 | PLAN_GENERATION |
| 包含工具 | query_rag, query_historical_cases |

#### 5.3.1 工具详细说明

| 工具名 | 输入 | 输出 | 业务场景 |
|--------|------|------|----------|
| query_rag | 自然语言查询 | Top-K 相关文档片段 + 来源引用 | 法规、预案、技术指南的语义检索，是知识查询的主入口 |
| query_historical_cases | 事故类型 + 关键词 | 匹配的历史案例列表 | 查找类似事故的处置经验和教训 |

#### 5.3.2 Prompt 片段要点

- query_rag 是知识查询的第一入口，优先于 query_historical_cases
- 当 query_rag 返回的法规不足以支撑方案时，再调用 query_historical_cases 补充实践经验
- 检索结果必须保留来源引用，写入 TaskState.knowledge_refs
- 模型应基于检索到的法规生成方案，而不是仅用自身知识

---

### 5.4 资源调度 Skill（Resource Dispatch）

| 属性 | 内容 |
|------|------|
| 名称 | resource_dispatch |
| 定位 | 盘点和调度可用的应急资源 |
| 激活阶段 | PLAN_GENERATION |
| 包含工具 | search_map_resources, search_nearby_pois |

#### 5.4.1 工具详细说明

| 工具名 | 输入 | 输出 | 业务场景 |
|--------|------|------|----------|
| search_map_resources | 事故点坐标 + 资源类型 | 附近资源列表（距离、值班人、联系方式） | 内部可调度资源查询，优先级最高 |
| search_nearby_pois | 经纬度 + POI 类型 | 公开 POI 列表 | 内部资源不足时的补充查询 |

#### 5.4.2 Prompt 片段要点

- 严格遵守"内部资源优先"规则：必须先调用 search_map_resources
- 只有当内部资源某个类型缺失时，才补充调用 search_nearby_pois
- 资源查询结果写入 TaskState.available_resources，并生成地图可视化数据
- 资源应按类型分组：医疗、消防、公安、物资、运力

---

### 5.5 风险评估 Skill（Risk Evaluation）

| 属性 | 内容 |
|------|------|
| 名称 | risk_evaluation |
| 定位 | 对已生成的应急方案进行多维度风险评估 |
| 激活阶段 | PLAN_EVALUATION |
| 包含工具 | risk_assessment |

#### 5.5.1 工具详细说明

| 工具名 | 输入 | 输出 | 实现特点 |
|--------|------|------|----------|
| risk_assessment | scenario(灾情描述) + plan(方案内容) | 综合得分、风险等级、各维度评分、改进建议 | LLM 套 LLM 实现，通过 prompt 约束输出 JSON 结构 |

#### 5.5.2 评估维度

风险评估工具应从以下维度对方案进行评分：

- **响应速度：** 资源到达时间是否合理
- **资源充足性：** 调度资源是否足够应对事件规模
- **法规合规性：** 方案是否符合相关法规要求
- **人员安全性：** 是否充分考虑了救援人员和群众安全
- **环境影响：** 对环境、周边交通的影响程度
- **可执行性：** 方案步骤是否清晰可操作

#### 5.5.3 评估结果处理规则

- 综合得分 >= 80：自动进入 OUTPUT 阶段
- 综合得分 60-79：通过人机协作 Skill 告知用户风险点，征求确认后进入 OUTPUT
- 综合得分 < 60：自动回退到 PLAN_GENERATION，带上改进建议重新生成方案
- 连续 2 次评估不通过：强制进入 WAITING_USER，请用户介入决策

---

### 5.6 人机协作 Skill（Human Collaboration）

| 属性 | 内容 |
|------|------|
| 名称 | human_collaboration |
| 定位 | 定义 Agent 与用户交互的规范，确保关键决策有人工参与 |
| 激活阶段 | 全部阶段（常驻） |
| 包含工具 | 无直接工具，通过 prompt 规范交互行为 |

#### 5.6.1 三种交互模式

**模式 A：信息补问** ——当缺少关键信息时触发

- 触发条件：IncidentInfo 中必填字段缺失，或工具返回的结果模糊需要确认
- 输出格式：明确告知缺什么、为什么需要、期望用户提供什么格式的信息
- 示例："您提到了高速公路事故，但未说明具体位置。请提供事故发生的具体路段（如'G4高速 K120 公里处'），以便我调度最近的救援资源。"

**模式 B：方案选择** ——当存在多个可行方案时触发

- 触发条件：存在 2-3 个可行方案，各有明显优劣势
- 输出格式：每个方案给出标签、核心思路、优势、劣势、适用场景
- 示例："基于当前灾情，我生成了两套方案：方案A（快速响应）优先调度最近资源，15分钟内到达，但资源规模有限；方案B（充分准备）调度更大规模资源，预计30分钟到达。请选择偏好的方案。"

**模式 C：确认征询** ——当方案存在风险或重大决策时触发

- 触发条件：风险评估得分在 60-79 区间，或涉及危化品、大规模人员转移等重大决策
- 输出格式：展示方案摘要 + 风险点 + 明确询问"是否确认执行"
- 示例："方案评估得分86分，主要风险点：救援路线经过拥堵路段，可能延误5-10分钟。是否确认执行该方案，还是需要我寻找替代路线？"

#### 5.6.2 紧急程度与交互策略

| 紧急程度 | 交互策略 | 说明 |
|----------|----------|------|
| critical | 先给默认方案，同时征求确认 | 不等待用户回复就开始执行，但并行征求确认 |
| high | 给出推荐方案 + 等待确认 | 有明确推荐，但等用户确认后再继续 |
| medium | 给出多个选项 + 等待选择 | 多个方案各有优劣，需要用户决策 |
| low | 完整展示分析 + 等待指示 | 有充足时间充分讨论 |

## 6. Agent Core 改造方案

### 6.1 当前问题

现有 Agent.chat() 的主要问题：

- CLI 和 Web 的工具调度逻辑不一致（CLI 只执行第一个工具，Web 执行全部）
- 工具定义全量加载，无动态筛选
- 无状态机，模型不知道当前处于什么阶段
- 无主动推进能力，只能被动响应

### 6.2 改造后的 Agent 主循环

新的 Agent 主循环伪代码如下：

```python
async def run(self, user_input: str):
    # 1. 初始化任务状态
    self.task_state = TaskState(phase='INTAKE')
    self.task_state.conversation_history.append(user_msg)

    # 2. 任务主循环
    while self.task_state.phase != 'OUTPUT_COMPLETE':
        # 2a. Skill Router: 根据当前 phase 加载工具和 prompt
        active_skills = self.skill_router.get_active_skills(
            self.task_state.current_phase)
        tools = merge_tools(active_skills)
        system_prompt = build_system_prompt(
            self.task_state, active_skills)

        # 2b. 调用模型
        response = await self.llm.chat(
            system_prompt, self.task_state.conversation_history, tools)

        # 2c. 如果有工具调用，执行并回结果
        if response.has_tool_calls:
            results = await self.execute_tools(response.tool_calls)
            self.task_state.update_from_tool_results(results)
            continue

        # 2d. 如果模型输出了状态过渡指令
        if response.has_phase_transition:
            self.task_state.transition(response.next_phase)
            continue

        # 2e. 如果需要用户输入，挂起等待
        if response.needs_user_input:
            self.task_state.phase = 'WAITING_USER'
            yield response.user_prompt  # 返回给前端展示
            return  # 等待下一次 user input

        # 2f. 最终输出
        if self.task_state.phase == 'OUTPUT':
            yield response.final_plan
            self.task_state.phase = 'OUTPUT_COMPLETE'
```

### 6.3 Skill Router 实现

```python
class SkillRouter:
    def __init__(self, skills_dir: str):
        self.skills = self._load_all_skills(skills_dir)

    def _load_all_skills(self, skills_dir) -> list[Skill]:
        """启动时扫描 skills/ 目录，加载所有 SKILL.yaml"""
        ...

    def get_active_skills(self, phase: str) -> list[Skill]:
        """根据当前 phase 返回应激活的 Skill 列表"""
        return [s for s in self.skills
                if phase in s.active_phases]
```

### 6.4 与现有代码的兼容策略

为了降低改造风险，建议采用渐进式迭代：

1. **第一步：** 保留现有工具实现代码（BaseTool 及其子类）不动，仅在其上层增加 Skill 封装层
2. **第二步：** 在 Agent Core 中增加 TaskState 和 SkillRouter，替换原有的固定工具加载逻辑
3. **第三步：** 统一 CLI 和 Web 的调度逻辑，共用同一个 Agent.run() 方法
4. **第四步：** 逐步将各工具迁移到对应的 Skill 目录下

### 8.3 技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 模型状态过渡判断不准 | 阶段跳转混乱，流程卡死 | 在 prompt 中明确定义过渡条件，增加阶段超时强制过渡机制 |
| 多 Skill prompt 合并后过长 | 抵消 Skill 机制的优势 | 控制单个 Skill 的 prompt 片段长度，设置上限 |
| 工具执行失败影响主流程 | 流程中断 | 每个工具调用加 try-catch，失败时返回降级结果而不是崩溃 |
| RAG 初始化慢 | 首次响应延迟高 | 将 RAG 初始化移到服务启动时，而不是每次查询时 |
| 长对话 token 溢出 | 模型丢失早期上下文 | 在状态过渡时压缩历史，只保留 TaskState 摘要 |

## 9. 可扩展性设计

### 9.1 新增 Skill 的标准流程

新增一个 Skill 只需要以下步骤，无需修改 Agent Core 代码：

1. 在 skills/ 目录下创建新目录
2. 编写 SKILL.yaml，定义名称、描述、激活阶段、包含工具
3. 编写 prompt.md，定义该 Skill 激活时注入的指导文本
4. 实现 tools.py，继承 BaseTool 实现具体工具
5. SkillRouter 会在启动时自动扫描并加载

### 9.2 潜在的新 Skill 方向

基于当前架构，未来可以较低成本地新增以下 Skill：

- **路径规划 Skill：** 救援车辆路径规划、替代路线推荐、预计到达时间
- **舆情监控 Skill：** 社交媒体、新闻媒体的实时舆情感知与处置建议
- **事后复盘 Skill：** 事件结束后生成复盘报告、总结经验教训
- **多事件协同 Skill：** 同时处理多个并发事件时的资源协调

## 10. 总结

本方案的核心变化是将系统从"全量工具塞入 prompt 的被动问答系统"升级为"基于 Skill 和状态机的任务驱动自主 Agent"。具体来说：

- 6 个 Skill 封装了全部能力，每个 Skill 独立可维护、可扩展
- TaskState 状态机驱动整个流程，确保模型始终知道"我在哪、该做什么"
- SkillRouter 动态加载工具，避免 token 浪费和注意力稀释
- 人机协作 Skill 确保关键决策有人工参与，同时根据紧急程度灵活调整交互策略
- 渐进式实施路线降低风险，保持与现有代码的兼容

这套架构让系统具备了"应急指挥场景的 Claude Code"的核心特征：接收任务后自主推进、按需调用工具、关键节点征询人工、最终交付完整方案。
